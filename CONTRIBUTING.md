## Why Contribute?

## Bug Reports

A good bug report helps us fix problems faster.

When reporting a bug, please include:
- A clear and descriptive title
- Flutter version and platform (Android, iOS, Web, Desktop)
- Steps to reproduce the issue
- Expected behavior vs actual behavior
- Code snippets or minimal reproducible examples (preferred)

Before opening a bug:
- Check if it has already been reported
- Test against the latest version of Fluttrap

---

## Feature Requests

Feature requests are welcome and encouraged.

When proposing a feature:
- Explain the problem it solves
- Describe how it fits into Fluttrap‚Äôs philosophy of speed and simplicity
- Provide examples or use cases if possible

Well-thought-out requests are far more likely to be accepted and implemented.

---

## Pull Requests

Pull requests are the heart of Fluttrap‚Äôs growth.

### Before submitting a PR:
- Open an issue or discussion for non-trivial changes
- Keep PRs focused on a single improvement or feature
- Follow existing code style and structure
- Add documentation or examples if applicable

### PR Best Practices:
- Write clear commit messages
- Avoid unrelated changes in the same PR
- Ensure code is formatted and analyzed (`flutter analyze`)
- Test changes locally before submitting

---

## Coding Guidelines

### Flutter & Dart
- Follow official Dart and Flutter style guidelines
- Prefer composition over inheritance
- Keep widgets small and reusable
- Avoid unnecessary abstractions
- Write readable, self-explanatory code

### Components & Extensions
- Components should be flexible and customizable
- Defaults should be sensible and production-ready
- APIs should feel intuitive to Flutter developers
